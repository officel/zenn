# tl;dr

- terraform をちゃんと書いていれば一度は目にする `required_version`
- 歴史のあるコードだと古いバージョンのまま放置していたり？
- アウトカムはないかもしれないけど適切なバージョン設定をしたいよね、という話

# required_version

- だいぶ前からある terraform block の中核機能（v0.8.0くらいからある）
- モジュールを作成する時は設定されているべき（強制力はないはず）
- tflint のルールでは必須扱い（なので事実上ちゃんと書くことになる）
- 歴史のあるコードの場合、最初に設定してそれ以降更新してないケースが多い（と思われる）
- インターナルなコードだったりするとアウトカムはまったくないんだけどそれなりに適切な値を設定したいよなぁとか思ったってワケ

# 基本的な考え方

- 使う機能に応じて適切な値が設定される **べき**
- 現状では指定範囲外のバージョンを使用する以外は問題にならない（`required_version = ">= 1.0"` なら 1.0 より古いバージョンは使えない。実行時にエラーになる）
- 今時わざわざそんなに古いバージョンを使用する必要もないので、たいていのところは 1.0 以上とか書いてあるはず
- ちなみに 2026-06-10 現在の terraform の最新バージョンは 1.15.6 です（さっきリリースがあったのでこの記事を書き始めた）
- 新しいバージョンの機能を使ってもワーニング等は出ないのであまり気にされていないかもしれないが、公開しているモジュール等では使用する機能に合わせて適切なバージョンを設定する必要がある（古いバージョンで実行してエラーになってもアレだからね）
- そういう意味では、インターナルなコードでは、古いバージョンを使用するケースはほぼないはずなので、問題にならないだけ
- それでいいの？本当に？

# 使用する主な機能とバージョンの対比

- 前述のとおり、`required_version` を適切に設定するためには、利用している機能に応じたバージョンを指定しておくのが望ましい
- というわけで、みんな使っているであろう機能のバージョンを洗い直してみた

| ブロック | 初出バージョン | 出典 |
| --- | --- | --- |
| `required_version` | v0.8.0 | PR [#10080](https://github.com/hashicorp/terraform/pull/10080) |
| `moved` block | v1.1.0 | [CHANGELOG v1.1.0](https://github.com/hashicorp/terraform/blob/v1.1/CHANGELOG.md) |
| `import` block | v1.5.0 | [CHANGELOG v1.5.0](https://github.com/hashicorp/terraform/blob/v1.5.0/CHANGELOG.md) |
| `removed` block | v1.7.0 | [CHANGELOG v1.7.0](https://github.com/hashicorp/terraform/blob/v1.7/CHANGELOG.md) |

- import block に for_each が使えるようになったのは v1.7.0
- ちなみに block の話であって、cli でのサブコマンドはそれぞれもうちょっと前についてるはず？

# というわけで

- 長期運用している terraform のコードベースで、import/moved/removed を使用するなら `required_version = ">= 1.7.0"` とするのが正しそう
- 古いバージョンのことなんか気にしないぜってんなら `>=1.15.0` でも問題ない
- とはいえ何度も書くけど、アウトカムが全然ないので、（パブリックなモジュールでもない限りは）書き直したからといって得られるものはせいぜい自己満足くらい

# まとめ

- `required_version = ">= 1.7.0"` を最低線にしましょう。もちろんやらなくても問題ありません。
- 最近 terraform のバージョンアップの追加機能があんまり興味を惹かれないので見直しがてら書いてみた、というワケ
