# tl;dr

- リポジトリに crlf が混入する問題についてどうしているかを聞きたい

# 経緯

- PR に crlf のファイルが混ざっていた
- GitHub ではわかりにくい
- AI のレビューが教えてくれてた
- 指摘するのもアレなんでしゅっと引っ張って修正（ファイル末尾改行もできていなかったので一緒に直して）push しておいた
- たぶんレビュイーの git の設定ができてない
- Windows ユーザーがチームにいて、今後も増減する可能性があり、git のレベルもまちまちである場合に、さてどうするかという話

# crlf は問題か

- 事実だけでいけば terraform のコードだったし今どき問題にはなりにくい（放置しても気持ち悪いだけで弊害はあまりない）
- 次に誰かが編集しようとした時に（lfに変換されれば）改行が差分になるだけ（それはそれでだるい）
- crlf じゃないといけないまたは lf じゃないといけない、とはルール決めしていない（ので少し悩んでいる）
- まぁ Mac ユーザーが多いし概ね lf なんですけれども

# 対応

## `.gitattributes`

- `* text=auto eol=lf` これ1行で足りる
- 今はチームのリポジトリにはどこにも置いてない（はず）
- ルール付けと一緒に置いておけば問題になることはない、はず？
- これのためだけにファイル置くのがちょっとやだなって

## `.editorconfig`

- `end_of_line = lf` で足りる
- 置いてあった。たぶん対応していないエディターで編集したのかも
- そりゃすり抜ける
- VS Code の設定で新規ファイルの改行コードを lf にしてもらうのもありかもな

## pre-commit 等のフック

- pre-commit なら標準のやつに `mixed-line-ending` ってゆうのがある
- 他のツールにもたいていある
- 自分では必要としてこなかったので入れてない
- 先日記事にしたように、チームに導入する準備もできてきたので入れてもいいんだけど、これも git config と同じで各自が設定しないとすり抜ける（CIで止められるけど）

## git config

```bash
git config --global core.autocrlf false
git config --global core.eol lf
```

- レビュイーの端末でやっといてって言えば済む
- 個人的にはこれを設定済（autocrlfだけ）
- うろ覚えなんだけどこれって `.gitattribute` とセットで eol が処理されるんじゃなかったっけ
- あと新規ファイルの時にダメだった気がする？

## 口頭説明または何もしない

- 本人に git の設定しておいてって言えばいったん問題ない
- global に設定しなかったら他のリポジトリで再発するけど
- 他の人が同じ問題を繰り返す可能性も残る

# 個人的には

- `.gitattributes` ファイルを置くのはちょっと邪魔だなって。リポジトリトップに毎度鎮座してほしくない
- `.editorconfig` はすり抜けちゃったので検討外
- `pre-commit` もすり抜けられるしCIできてないリポジトリも多くある
- `git config` をやってもらうなら口頭説明になる
- というか、この話が必要なら `core.ignorecase` とか `core.filemode` の話もしないといけなくない？
- とか思っていて、だけど実際には自分の dotfiles 公開してるし説明してるし、取り込みが終わってないだけなんだよね
