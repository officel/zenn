# tl;dr

- GitHub を使って旅行計画を整理して記録しておくと便利なんすよって話
- 家族旅行や仲間内、人数がある程度必要で幹事をする時の参考に

# ツールの選定

- 参加人数の多い旅行の計画を、どこにどのように残してどう活用するか
- 事前にPCでもスマホでも検索・整理できて、現地でもスマホで確認できる必要がある
- 使い慣れたツールが一番
- ADR的に比較検討記事にしようかと思ったんだけど、好みで決めたから比較できなかったｗｗ
- というわけで GitHub を使って家族旅行の計画を整理した方法を記事にしたというワケ

# 下ごしらえ

GitHub のでスマホアプリをインストールしておく。

- プライベートな情報だし、プライベートリポジトリを用意する
- zenn の読者はプライベートな TIL リポジトリのひとつやふたつ持ってるよね？
- 専用のリポジトリを作ってもよい（使い方の練習だと思えば）
- あとで参照する時に手間がかかるので、既存のプライベートリポジトリがあればそのまま使うほうがいい

# Discussions

GitHub Discussions は計画を練るために使う。

- カテゴリーを1つ用意
- どこへ行くか（どうやっていくか）
- 日程調整（宿泊するならどこに泊まるのか）
- 譲れないものは何か（あれがしたい、これが見たい、それが食べたいなど）
- 邪魔なものは何か（該当日程中に近隣でコンサートがあって宿が取れないなど、気にしておくことがないか）
- それぞれの優先順位などなど
- スレッドも1つのほうが後で楽だけど、時間と検討に余裕があるなら別でも

# Projects

GitHub Projects は計画を整理するために使う。

- 出発から帰宅までの動きをアイテムにして登録する（まだissueにはしない）
- 公共交通機関を利用するなら路線毎に書き出す（料金が別なものは個別にするくらいでちょうどいい）
- 行く場所、すること、泊まるところ、もちろん帰りの分までなるべく書き出す
- ある程度の合意が取れるまではアイテムのままにしてく（issue化はあとで）
- ビューを作って整理し、日程順に並べておくことで検討のヌケモレも防止できる
- アイテムのタイトルはわかりやすく並べやすいものにする
- 日付、往路か復路か、路線名、施設名など

# Issues

GitHub Issues は計画を実行するために使う。旅行計画では実現可能性や予約、その記録などを残す。

- だいたいの意思決定が終わったら Issue にする
- 親 Issue は "日程：計画名" のようなわかりやすい名前にする。旅行が終わるまでの間、参照する中心とする
- 中身は空でも大丈夫だし、ディスカッションへのリンクを入れてもよい
- Projects を使用せず、やるべきことリストをここで作成してもよい。まとめて sub-issue 化できる
- その場合 sub-issue の並び順が作成時点のものになるので、あとでちょっと不満が出るかもしれない
- Projects のアイテムを sub-issue として親 Issue にぶら下げる
- この場合 sub-issue 化した順に並ぶ（きちんと計画していれば不満はない、はず）
- 費用をまとめるために "費用メモ" sub-issue を用意すると精算が楽になる。計画外の支出もコメントなどで残しておけば振り返り時に役に立つ

# Issue を処理する

- 交通機関の予約（往復・路線別の sub-issue にしておき、それぞれ予約内容を転記しておく。時間の管理が重要）
- 宿の予約（HPと住所、電話番号、予約した確認メールの内容を転記しておく）
- 施設のHP（移動手段、営業時間、料金等を転記しておく）
- 旅先での移動手段の確認（今回はバスになりそうなので時刻表など）
- プランのブラッシュアップ（時間が不足しそうとか予算が厳しそうなどよくある）
- などなどを issue を通じて処理する
- 予約の結果はメール等で届くと思うが、前述のとおり、各 Issue に丸ごと転記しておくと便利
- 処理が終わった sub-issue は閉じてしまえば邪魔にならない（親 Issue からいつでも見ることができる）

# 行程表

タスクの処理をしながら親 Issue を行程表として整理する

- Issue はマークダウンで書けるので、日付別ヘッダーにする
- 表形式だと修正が手間なので、リストにするのが楽
- 使うかどうかは別として、チェックリスト `- [ ]` にしておく
- 各行程は `HH:MM: 内容` のような形にそろえておく
- 同施設内での行程はインデントしておく
- 最終的にこの親 Issue を見れば行程がわかり、関連する sub-issue でタスク状況がわかり、必要な準備が整っているかを確認できるようにする

# チェックリスト

自分用 "チェックリスト" を作っておくと次から楽できる。

- 荷物の事前送付が必要かどうか
- 出発帰宅時刻が適切か（通勤時間に大荷物・大人数で電車には乗れない）
- 時刻表の平日、土日ダイヤの見間違え等はないか
- 休止対策、天候対策はしてあるか（臨時休業や悪天候などの場合にプランBが用意してあるか）
- 遅延対策はしてあるか（予定がずれた時にどうするか検討してあるか）
- 時刻表、料金表の変更予定はないか（JRは事前購入分は値上げしないので早めの購入が吉。温泉地などでは入浴税が変わったりする。今回は伊勢鳥羽に行ったが来月から入浴税が値上がりするようだ。つまりどっちもセーフだったｗ）

# その他の TIPS

- 時刻表類はスクリーンショットを取って貼っておくと毎回調べないで済む。不要になったら削除しても問題ない
- 旅先の天気予報ページへのリンクを貼っておくと毎回調べないで済む
- （前述済） "費用メモ" があると、どれだけかかるのか、かかったのかを記録しておける
- （前述済）日程中の別の何かには注意する。交通費宿泊費が驚くほど違う。混雑回避のためにも行き先を決める時によく調べる
- "振り返り" を最後に作って、旅行後に記録を残せるようにする。主にヌケモレと計画と実行間のズレを記録しておいて次に活かす

# 雑記

- 旅行計画は人数多いと大変
- 家族旅行のためにひとりプランニングはつらたんだった
- 几帳面な性格をフルに発揮して完璧な計画とその実行を実践したのでまぁ満足
- システム開発するのと同じだし、練習とか訓練のつもりでやって見るといいよ！

# さらに雑記

- 旅行行く前にこの記事の下書きをしていて、こういう旅行計画用のアプリあったらいいなぁとか思った
- 個人開発のネタにもいいんじゃない？って思ったんだけど、当然のごとく腐るほどあったｗ
- 国内では旅行業法があるので予約系はほぼ無理
- ホテルの予約は基本的に直接が一番高効率だと思ってる
- 旅のしおり系、スケジュール、vlog類も出揃ってる感じ
- 大手が多数参入してるので新規参入では差別化が必要
- AIなんちゃらが多数出てるのは時代かしらね
- 数は力だと思うけど、そもそも稀にしか旅行しない人からするとアプリの良し悪しを調べる機会もないんだ。。。
