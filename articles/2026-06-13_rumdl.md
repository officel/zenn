# markdown の lint

- markdown 関係のリンターやフォーマッターは数がとても多い
- CommonMark や GFM はもちろん、MkDocs 等のツール類に向けた拡張など、markdown の方言が数多いことに原因がある
- あちらを立てればこちらが立たずなのはどこも同じ
- フレーバーとしてターゲットにする方言の種類に合わせて対応ができるとよさげ
- 個人やチームで利用するなら、必要に応じて調整しやすいほうがうれしい
- No Dependant つまり単一バイナリだとうれしい
- Zero Configuration つまりデフォルト設定である程度動くとうれしい

# pymarkdown 代替となる Rust/Go 製 Markdown リンターの調査と意思決定レポート

現在ご利用中の `pymarkdown` の代替として、高速化を目的とした Rust または Go 製の Markdown リンターを20個調査しました。ご要望の通り、本家である `markdownlint` (Node.js版) との互換性、高速性、活発な開発状況（2026年の更新）、リリースノートの充実度、および `pre-commit` への対応状況を基準に評価し、ランキング形式でまとめました。

## 評価基準とスコアリング

各ツールは以下の基準で評価・スコアリングしています。

1. **markdownlint 互換性**: 本家 `markdownlint` のルールセットとの互換性が高いものを高く評価。
2. **高速化**: Rust/Go によるネイティブコンパイルの恩恵を受け、高速に動作するものを高く評価。
3. **活発な開発**: 2026年に入ってからも更新や活動が見られるものを評価。
4. **リリースノート**: 変更履歴やリリースノートがしっかりと管理されているものを評価。
5. **pre-commit 対応**: `pre-commit` フックが公式に提供されている、または言及されているものを評価。

## 💡 意思決定のための推奨事項 (Decision Making)

ご要望の「`markdownlint` ベースであること」「高速化」「活発な開発」「リリースノートの充実」「`pre-commit` 対応」をすべて満たすツールとして、以下のトップ3を強く推奨します。

### 🥇 1. rumdl (Rust)

* **推奨理由**: 現在最も勢いがあり、Pythonの `ruff` にインスパイアされた超高速リンターです。本家 `markdownlint` との互換性が非常に高く（97.2%）、75のルールをサポートしています。
* **パフォーマンス**: 本家と比較して30〜100倍の高速化を実現しています。
* **エコシステム**: `pre-commit` フックが公式に提供されており、VS Codeなどのエディタ拡張も充実しています。

### 🥈 2. mado (Rust)

* **推奨理由**: `markdownlint` のルールを忠実にサポートすることを目指して開発されているRust製リンターです。
* **パフォーマンス**: 本家と比較して約49〜60倍の高速化がベンチマークで示されています。
* **エコシステム**: `.pre-commit-hooks.yaml` がリポジトリに含まれており、CIへの組み込みが容易です。

### 🥉 3. goldmark-lint (Go)

* **推奨理由**: Go言語の標準的なMarkdownパーサーである `goldmark` をベースに、`markdownlint-cli2` のルールをGoに移植したプロジェクトです。
* **特徴**: 設定ファイル（`.markdownlint.yaml` 等）のフォーマットも本家と互換性を持たせており、移行コストが非常に低いです。Go環境をメインにしている場合に最適です。

### 結論

`pymarkdown` からの移行において、**最もシームレスかつ劇的な速度向上**を見込めるのは **`rumdl`** です。もしGo言語のエコシステムに統一したい場合は **`goldmark-lint`** が最良の選択肢となります。まずはこの2つのいずれかを試験的に導入し、既存のルール設定がどの程度そのまま機能するかを確認することをお勧めします。

# rumdl を使ってみた

## .pre-commit-config.yaml に乗せる

- 既存のやつもしばらく併用
- 最初は `--fix` と `rumdl-fmt` はオフにしておいて調整すること

```yaml:.pre-commit-config.yaml
  - repo: https://github.com/jackdewinter/pymarkdown
    rev: 5a91b386476b633c4a594951397139aa5ab45a78  # frozen: v0.9.38
    hooks:
      - id: pymarkdown
        args:
          - --config=.config/.pymarkdown.yaml
          - scan

  - repo: https://github.com/rvben/rumdl-pre-commit
    rev: 65796824868c4b3bea1743498325a5b87350c2e8  # frozen: v0.2.15
    hooks:
      - id: rumdl
        args: ["--fix"]
      - id: rumdl-fmt
```

## バグというかドキュメントのミスを見つけた

- [Why don't we update the documentation for the `pre-commit integration` section? · Issue #666 · rvben/rumdl](https://github.com/rvben/rumdl/issues/666)
- ちょっとテストしてみたら v0.2.14 のアクション `id: rumdl` が `--fix` されるようになっていた
- 本体の README に書かれていた内容と齟齬があったので、上記の issue を立てた
- 数時間で確認してもらえて修正された（v0.2.15になった）
- こういうのもツール移行をする時の醍醐味って言うのかしらね
- ツール移行駆動学習なかなかいいと思うの

## 設定ファイルを積み上げるまでのあれこれ

- 前述のとおり、初期では期待と違って大量にファイルが fix されてしまった
- 約160ファイル2000近いfixが秒で終わるくらい早かった
- とりあえず fix はユーザーの選択で行われるべき（勝手にやっちゃダメ）だと思っているので、issue にも書いたけど `entry:` を上書きする方法で対応しておいた（v0.2.15で解決しているのでもう気にしなくていい）
- こういう時に pre-commit(prek) の扱いをちゃんとやってきてよかったなってｗ
- pymarkdown の設定ファイルからの移行ツールはないようだったので、AI に移行させたら嘘ばかりだったので手で書き直した
- 外部のAIを使うんじゃなくて、DeepWiki とか Notebook とか使ったほうがよかったかもしれない
- 個人用設定だし手で積み上げるほうが最終的に勉強になってよかったね
- 既存の markdown ファイルに適用してみて調整が必要なので、決め事も含めて設定ファイルを運用していくのが肝要
- プライベートのリポジトリであれこれやって調整して、このブログ用のリポジトリに持ってきてもなお調整が要る
- markdown の lint は奥が深い

## 設定について

- 設定ファイルは命名により自動読み込みができていて、`.config/rumdl.toml` でいける（個人的にとてもうれしい）
- 設定例も含めてドキュメントが充実しているので、とてもわかりやすい
- global テーブル（toml書いてない人にテーブルとか伝わるか？ `[global]` のこと。ini ファイルでいうセクションみたいな）の `exclude` は個別のルール設定とバッティングすると見にくいので使わないほうがいいかなと思った
- 個別のルールテーブルで使用可否が設定できるので、コメントも含めて個別にきちんと書くのがわかりやすい
- global テーブルに `include` と `exclude` を書けるので、`.pre-commit-config.yaml` 側はすっきりしていい
- 詳細はこの記事のリポジトリを見て
