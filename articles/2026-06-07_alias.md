# 自分用 alias のプラクティス

- 使う分だけ設定する（試しに設定してみるのはよし）
- 安易にコピペして満足しない（使わない設定が残る。。。）
- ポータビリティを気にしない（自分の環境ではないところに自分の環境を押し付けない。郷に入っては郷に従えということ）
- シェル関数との使い分け（個人的にはできるだけ使用しない方針）
- 破壊的コマンドの安全策は必要になるまでしない（必要ならする）
- 定期的な見直しの実施
- 設定チェック

  - alias に alias を重ねない
  - 既存のコマンド類とバッティングしないか（`type` 等で確認）
  - 一貫性を確保する
  - 文脈から読み取りやすいaliasになっているかを確認

- 設定方針

  - よく使うコマンドだけaliasにする（1文字化する）
  - 普段使わないコマンドはaliasにしない
  - 覚えていないオプション類はできるだけロングオプションを使用して設定
  - 覚えているオプションを短くするだけの場合はショートオプションを使用してよい

# 最近お気に入りの設定方針だけ説明を書く

- 普段遣いするコマンドこそ alias にしてタイピング数を減らすことで時短に繋げる
- a から z までの1軍 alias を大事にする
- 覚えてない、覚えにくいオプションやサブコマンドはロングオプションを alias に記載しておくことで思い出しやすくする
- そのために `alias aliasxx='xx'` を用意する
- というわけで `git` の alias の例

```shell
# git は1軍なので g に設定。後続する alias では省略せずに git を使うこと
alias g='git'

# 覚えているオプションはショートで書く（-a はたいていのコマンドで add か all ）
alias gba='git branch -a'

# 既存のコマンドとぶつからないようにするのと同時に類似のオプションとの一貫性を保つ
alias gdi='git diff' # ライブラリとのバッティング予防と、他の diff オプションとの兼ね合い

# ロングオプションを使用することで見返した時に理解しやすい
alias gdw='git diff --no-index  --word-diff-regex="."'

# よく使うコマンドの流れを揃えておくと使いやすい
alias gr='git fetch --prune --all && git checkout origin/HEAD && git plog --all -10 && git status'
alias cr='cd "$(git rev-parse --show-toplevel)"'
# ↑の例だと cr して gr するとリポジトリトップにいてリモートと同期が終わっていて差分が表示される


# git は g にしたので、aliasg とすることで一覧を出す（このためにも alias を alias しないようにする）
alias aliasg='alias | grep git'
```

# やめたこと

- コマンドやオプションを覚えていないから毎回ちゃんとタイプする

  - git 覚えはじめのころ思っていた
  - 結局覚えなかったし、alias を設定してタイプ数を減らして時短した上で発生した余剰時間を他のことに使うほうが有意義だった
  - 今では alias にして、何を使って何を使ってないかを明確にしたほうが学習効率があがると思っている

- 他人の設定の丸コピー

  - これは自分でこうだっていう設定ができるようになったおかげ
  - 一家言持てるようになるまでは丸コピーのほうが手っ取り早い
  - よくブログ記事に出ている設定例を疑ってかかって、より自分向きの設定をできるようになるとよい

- 機械的設定

  - 探すといろいろ出てくると思うんだけど、kubectl とか git のオプションや設定類を機械的に並べてアクロニムで alias にするリポジトリがあって、これでなんでもできるぜ！って思ってた時期があったんだけどやめた
  - 使わないやつは使わないし、アクロニム化するほどオプション類を覚えていないので結局使えないというか、必要な分を検索してみないとわからないという状態になりがちだったため
  - 多少不格好でも自分で調べて設定して使う分だけ育てたほうが価値があるかなって

- バッティングで悩まない

  - 以前は `terraform` が一軍の `t` だった
  - `task` を使うようになって、`t` を明け渡すのに悩んだ
  - 今は `alias tf='terraform'` 、`alias t='task'` にしている
  - 一日のタイプ回数（使用頻度）とわかりやすさ
  - dotfiles リポジトリで git 管理しているならちょいちょい書き直してもたいした手間じゃない
  - ＝自分の好みになるまで何度もやり直せばいいだけ

# tl;dr

- コマンドaliasは生産性を高める強力なツールだが、無秩序な設定は混乱と事故の元。
- 「使う分だけ」「既存コマンドと衝突しない」「一覧性を保つ」を基本方針に、規律を持って活用しよう。

# はじめに：なぜalias設定に「規律」が必要なのか

日々の開発や運用作業において、シェルコマンドのalias（エイリアス）は私たちの生産性を劇的に向上させてくれる強力な味方です。しかし、その便利さゆえに、何も考えずに設定を増やしていくと、やがては「aliasの墓場」と化し、かえって作業効率を下げたり、思わぬ事故を引き起こしたりする原因にもなりかねません。

本記事では、コマンドaliasを効果的に活用するための「設定方針」について、私の考えと、これまでの経験から得られたベストプラクティスを共有します。無秩序なalias設定に終止符を打ち、よりスマートなシェル環境を構築するための一助となれば幸いです。

# 基本方針（Do）：賢いaliasの育て方

## 1. 厳選主義：本当に「使う分だけ」をaliasにする

aliasは、あなたが頻繁に、かつ無意識にタイプするコマンドに対してのみ設定すべきです。指が自然と動くような、まさに「身体の一部」となるコマンドに絞り込みましょう。

例えば、`git checkout` のような長くてタイプミスしやすいコマンドを `gco` と短縮するのは非常に効果的です。しかし、月に一度しか使わないようなコマンドにまでaliasを設定するのは、覚える手間と、いざという時に思い出せないリスクを考えると、費用対効果が低いと言えます。

**ポイント:**

- **「指が覚えているか」を基準にする**: 頻繁に使うコマンドに限定し、記憶の負荷を減らす。
- **長いコマンドの短縮**: タイプミスを減らし、入力速度を向上させる。

## 2. 衝突回避：既存のコマンドや将来のツールとぶつからないように注意する

aliasを設定する際に最も注意すべき点の一つが、既存のコマンド名や、将来的に導入する可能性のあるツール名との衝突です。意図せずシステムコマンドの挙動を変えてしまったり、新しいツールが使えなくなったりする可能性があります。

aliasを設定する前には、必ず `type <コマンド名>` コマンドで、その名前が既にシステム内で使われていないかを確認する習慣をつけましょう。例えば、`d` を `docker` のaliasに設定したとして、後日 `d` という別のCLIツールを導入しようとした際に、衝突して使えない、といった事態を避けることができます。

```bash
# aliasを設定する前に確認
$ type ls
ls is aliased to `ls --color=auto'

$ type mycommand
mycommand not found
# これならaliasに設定しても安全
```

**ポイント:**

- **`type` コマンドで事前確認**: 既存コマンドとの衝突を避ける。
- **将来的なツールとの衝突も考慮**: 短すぎる、汎用的な名前は避ける傾向にある。

## 3. 自己ドキュメント化：いつでもaliasリストを出せるようにする

設定したaliasは、時間が経つと忘れてしまうものです。そのため、いつでも自分のaliasリストを簡単に確認できる仕組みを用意しておくことが重要です。

単に `alias` コマンドを実行するだけでなく、例えば `help-alias` のような専用のaliasやシェル関数を作成し、`~/.bashrc` や `~/.zshrc` に設定しているalias定義ファイルの内容を整形して表示できるようにすると、より便利です。

```bash
# 例: alias定義ファイルの内容をgrepでフィルタリングして表示
alias help-alias='grep "^alias" ~/.zshrc | sort'
```

**ポイント:**

- **一覧性の確保**: 忘れてもすぐに確認できる仕組みを作る。
- **カスタムヘルプの作成**: `grep` や `awk` を使って見やすい形式で表示する工夫。

# アンチパターン（Don't）：避けるべきalias設定

## 1. 機械的な設定：内容を理解しないままコピペしない

インターネット上には「最強の.zshrc」といった記事が多数存在します。これらを安易にコピペして自分の環境に適用するのは非常に危険です。あなたが理解していないaliasが設定されていると、意図しないコマンドが実行されたり、セキュリティ上のリスクを招いたりする可能性があります。

aliasは、あなたがそのコマンドの意図と挙動を完全に理解した上で、自身の作業フローに合わせてカスタマイズするべきものです。

**ポイント:**

- **理解なきコピペは危険**: 意図しない挙動やセキュリティリスクを招く。
- **自身の作業フローに合わせたカスタマイズ**: 自分で設定し、自分で管理する意識を持つ。

## 2. 多段エイリアス：aliasにaliasを重ねない

`alias a='b'` と設定し、さらに `alias b='c'` といったように、aliasが別のaliasを参照する「多段エイリアス」は避けるべきです。これはデバッグを非常に困難にし、aliasの依存関係を複雑化させます。

元のコマンド `c` を変更した際に、`a` や `b` の挙動がどう変わるのかが分かりにくくなり、「隠れた依存性」の問題を引き起こします。aliasは常に、最終的に実行される元のコマンドを直接参照するようにしましょう。

**ポイント:**

- **デバッグの難易度上昇**: 挙動の追跡が困難になる。
- **隠れた依存性の問題**: 変更時の影響範囲が不明瞭になる。

# 発展編：エイリアスを卒業してシェル関数にするタイミング

aliasは単純なコマンドの短縮には最適ですが、引数を伴う複雑な処理や、条件分岐、複数のコマンドを順次実行するような場合には不向きです。このようなケースでは、シェル関数（Shell Function）の利用を検討すべきです。

エイリアス vs シェル関数

| 特徴 | alias | シェル関数 |
| :--- | :--- | :--- |
| **用途** | 単純なコマンドの短縮、オプションの固定 | 複雑な処理、引数の利用、条件分岐、複数コマンドの実行 |
| **引数** | 扱いにくい（`$1` などが展開されない） | 容易に扱える（`$1`, `$2` など） |
| **柔軟性** | 低い | 高い |
| **例** | `alias ll='ls -alF'` | `my_func() { mkdir -p "$1" && cd "$1"; }` |

例えば、新しいディレクトリを作成してすぐにそのディレクトリに移動する `mkcd` のようなコマンドは、引数を受け取る必要があるため、シェル関数として定義するのが適切です。

```bash
mkcd() {
  mkdir -p "$1" && cd "$1"
}
```

**ポイント:**

- **引数や複雑なロジックが必要な場合**: シェル関数を検討する。
- **役割の明確化**: aliasは短縮、関数は処理の自動化。

# ポータビリティへの配慮：他環境での作業とalias

aliasに過度に依存しすぎると、SSHで接続したリモートサーバーや、新しい開発環境など、aliasが設定されていない環境で作業する際に「指が動かない」という問題に直面することがあります。これは、aliasが「個人の生産性向上」に特化したツールであることの裏返しでもあります。

この問題へのアプローチはいくつか考えられます。

- **最小限の共通aliasセット**: どの環境でも共通して使うごく基本的なaliasのみを定義し、それ以外は環境ごとに調整する。
- **aliasなしでの作業練習**: 意識的にaliasを使わない時間を設け、元のコマンドをタイプする練習をする。
- **dotfiles管理**: alias設定を含むdotfilesをGitなどで管理し、新しい環境に簡単にデプロイできるようにする。

チームで作業する環境では、共通のaliasやシェル関数を定義し、それを共有する仕組みを導入することも有効です。

# 「破壊的コマンド」の安全策としてのalias

`rm` コマンドのように、誤って実行すると重大な結果を招く可能性のある「破壊的コマンド」に対して、安全策としてaliasを設定するケースがあります。

例えば、`alias rm='rm -i'` と設定することで、ファイルを削除する際に毎回確認プロンプトが表示されるようになります。これは、既存のコマンドの挙動を変更する行為であり、前述の「既存のtypeとぶつからない」という方針とは一見矛盾するように見えます。

しかし、この種のaliasは「事故防止」という明確な目的があり、そのメリットがデメリットを上回ると判断される場合に限り、例外的に許容されるべきだと考えます。ただし、チームで作業する環境では、このようなaliasの導入は慎重に検討し、チーム内で合意形成を行うことが不可欠です。

**ポイント:**

- **明確な目的（事故防止）がある場合**: 例外的に許容される。
- **チームでの合意形成**: 複数人で作業する環境では特に重要。

# まとめ：道具を使いこなすための「規律」

コマンドaliasは、適切に管理すればあなたの生産性を飛躍的に向上させる素晴らしい道具です。しかし、その力を最大限に引き出すためには、明確な「規律」と「方針」を持って設定・運用することが不可欠です。

本記事で提示した「使う分だけをaliasにする」「既存のtypeとぶつからないように注意する」「いつでもリストを出せるようにする」という基本方針と、避けるべきアンチパターン、そしてシェル関数との使い分けやポータビリティへの配慮といった発展的な視点が、あなたのシェル環境をより快適で安全なものにするための一助となれば幸いです。

あなたのシェルは、あなた自身が育てる庭のようなものです。愛情と規律を持って、最高の庭を築き上げましょう！
